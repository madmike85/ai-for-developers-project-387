# Lighthouse CI - Автоматическая проверка производительности

## Обзор

В проекте настроена автоматическая проверка производительности веб-приложения с помощью **Lighthouse CI**. Проверка запускается:
- **Автоматически**: каждый день в 7:00 UTC (10:00 МСК)
- **Вручную**: через интерфейс GitHub Actions

## Что проверяется

Lighthouse CI анализирует приложение по следующим категориям:

- 🚀 **Performance** (производительность) - скорость загрузки, интерактивность, стабильность
- ♿ **Accessibility** (доступность) - удобство для пользователей с ограниченными возможностями
- ✅ **Best Practices** (лучшие практики) - современные веб-стандарты, безопасность
- 🔍 **SEO** (оптимизация для поисковых систем) - видимость в поисковиках
- 📱 **PWA** (Progressive Web App) - возможности офлайн-работы и установки

## Как просмотреть отчеты

### 1. Найти выполненный workflow

1. Перейдите на вкладку **Actions** в репозитории GitHub
2. В левой панели выберите workflow **"Lighthouse CI"**
3. Выберите нужный запуск из списка (показывает дату и время)

### 2. Скачать артефакты

1. Прокрутите страницу выполнения workflow вниз до раздела **"Artifacts"**
2. Найдите артефакт с именем `lighthouse-reports-{номер_запуска}`
3. Нажмите на него для скачивания (ZIP архив)

### 3. Открыть отчет

1. Распакуйте скачанный ZIP архив
2. Откройте файл `*-report.html` в браузере
3. Изучите детальный отчет с рекомендациями

### 4. Просмотр краткой сводки

На странице выполнения workflow прокрутите вниз до раздела **"Summary"** - там отображается краткая информация о проверке.

## Как запустить вручную

1. Перейдите на вкладку **Actions** в репозитории GitHub
2. В левой панели выберите workflow **"Lighthouse CI"**
3. Нажмите кнопку **"Run workflow"** (справа)
4. Выберите ветку (по умолчанию `main`)
5. Нажмите зеленую кнопку **"Run workflow"**
6. Дождитесь завершения (обычно 3-5 минут)

## Интерпретация результатов

### Оценки Lighthouse

Каждая категория оценивается по шкале 0-100:

- **90-100** 🟢 - Отлично
- **50-89** 🟡 - Требует улучшений
- **0-49** 🔴 - Плохо

### Ключевые метрики Performance

- **FCP** (First Contentful Paint) - время до первого отображения контента
- **LCP** (Largest Contentful Paint) - время до отображения основного контента
- **TBT** (Total Blocking Time) - время блокировки главного потока
- **CLS** (Cumulative Layout Shift) - стабильность визуального отображения
- **SI** (Speed Index) - скорость визуального отображения

### Что делать с результатами

1. **Просмотрите отчет** - откройте HTML файл и изучите оценки
2. **Найдите проблемы** - Lighthouse показывает конкретные рекомендации в разделе "Opportunities" и "Diagnostics"
3. **Приоритизируйте** - сфокусируйтесь на проблемах с наибольшим влиянием (показано в миллисекундах экономии)
4. **Создайте задачи** - заведите issues в GitHub для важных улучшений
5. **Отслеживайте прогресс** - сравнивайте отчеты день ото дня

## Локальный запуск (для разработчиков)

Для тестирования Lighthouse CI локально перед коммитом:

```bash
# 1. Перейдите в директорию проекта
cd code

# 2. Соберите приложение
npm run build --workspace=web

# 3. Запустите preview сервер (в отдельном терминале)
npm run preview --workspace=web -- --port 4173

# 4. Установите Lighthouse CI глобально (один раз)
npm install -g @lhci/cli

# 5. Запустите проверку (из корня проекта)
cd ..
lhci autorun

# 6. Откройте отчет
open .lighthouseci/*-report.html
```

## Конфигурация

### Файлы конфигурации

- **`.lighthouserc.json`** - настройки Lighthouse CI (URL, количество запусков, категории)
- **`.github/workflows/lighthouse-ci.yml`** - GitHub Actions workflow (расписание, steps)

### Изменение расписания

Чтобы изменить время запуска, отредактируйте cron выражение в `.github/workflows/lighthouse-ci.yml`:

```yaml
schedule:
  - cron: '0 7 * * *'  # Каждый день в 7:00 UTC
```

Примеры других расписаний:
- `'0 6 * * 1-5'` - каждый будний день (Пн-Пт) в 6:00 UTC
- `'0 9 * * 1'` - каждый понедельник в 9:00 UTC
- `'0 */6 * * *'` - каждые 6 часов

Используйте [crontab.guru](https://crontab.guru/) для помощи с cron выражениями.

### Изменение количества запусков

Lighthouse CI запускает проверку 3 раза и берет медианное значение для точности. Чтобы изменить:

```json
// .lighthouserc.json
{
  "ci": {
    "collect": {
      "numberOfRuns": 5  // Увеличить до 5 для большей точности
    }
  }
}
```

⚠️ **Примечание**: Больше запусков = дольше выполнение workflow

## Хранение отчетов

- **Хранилище**: GitHub Actions Artifacts
- **Срок хранения**: 90 дней
- **Формат**: HTML (для просмотра) + JSON (для автоматизации)
- **Расположение**: Скачиваются как ZIP архив из интерфейса Actions

После 90 дней артефакты автоматически удаляются. Если нужно сохранить отчет надолго - скачайте его локально.

## Частые вопросы

### Почему workflow запускается локально, а не на production URL?

Это позволяет проверять качество кода независимо от инфраструктуры и хостинга. Проверяется production build приложения в чистом окружении.

### Можно ли проверять staging или production URL?

Да, измените URL в `.lighthouserc.json`:

```json
{
  "ci": {
    "collect": {
      "url": ["https://your-production-url.com"]
    }
  }
}
```

И удалите steps запуска preview сервера из workflow.

### Workflow упал - это нормально?

Workflow настроен с `continue-on-error: true` для Lighthouse step, поэтому он не должен падать. Если падает на других step - проверьте логи для диагностики.

### Как добавить пороговые значения?

Отредактируйте `.lighthouserc.json` для добавления assertions:

```json
{
  "ci": {
    "assert": {
      "assertions": {
        "categories:performance": ["error", {"minScore": 0.8}],
        "categories:accessibility": ["warn", {"minScore": 0.9}],
        "categories:best-practices": ["warn", {"minScore": 0.85}],
        "categories:seo": ["warn", {"minScore": 0.9}]
      }
    }
  }
}
```

⚠️ **Внимание**: С assertions workflow будет падать при низких оценках.
